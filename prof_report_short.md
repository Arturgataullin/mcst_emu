# Короткий отчет по профилированию

Профилирование выполнялось на сборке `RelWithDebInfo`. Сначала проверялись отдельные
benchmark-и, потом по `callgrind` стало видно, что в разных примерах повторяются
одни и те же проблемы: чтение инструкции через общий `Memory::read32()` и поиск
блока разреженной памяти.

За baseline использовалась папка `profile_res_before`.

## Baseline

Основные данные до оптимизаций:

| benchmark | time | instructions |
| --- | ---: | ---: |
| `prof_arith_loop` | 3.458 s | 59.997 млрд |
| `prof_memory_loop` | 1.082 s | 18.188 млрд |
| `prof_branch_loop` | 1.665 s | 29.194 млрд |
| `prof_call_loop` | 0.971 s | 17.366 млрд |
| `prof_ajmp_loop` | 5.904 s | 46.106 млрд |
| `prof_page_alloc_loop` | 1.194 s | 17.113 млрд |

В `callgrind` даже для `prof_arith_loop` было видно, что много времени уходит в
`Memory::read32()`: `step()` занимал 33.00%, `read32()` - 26.29%,
`execute()` - 25.61%. Это означало, что fetch инструкции слишком дорого проходит
через общий интерфейс RAM.

## Что было сделано

1. После отдельных замеров для `prof_arith_loop` рассматривался вариант хранить
   программу отдельно как `programWords_` или сразу как массив `DecodedInstruction`.
   Он дал хороший результат на арифметическом benchmark-е: 3.458 s -> 2.494 s.
   Но от него отказались как от основного решения. По ТЗ программа находится в
   общей памяти вместе с данными, а отдельный массив кода хуже соответствует этой
   модели. Кроме того, появляется второй источник истины для кода: если программа
   будет изменять память с кодом, отдельный массив нужно синхронизировать с
   `Memory`.

2. Для доступа к разреженной памяти был добавлен кэш последнего блока чтения и
   записи. На `prof_memory_loop` время уменьшилось с 1.082 s до 0.792 s.

3. Для fetch был добавлен облегченный путь `Memory::readInstruction32()`. Он не
   делает лишние проверки общего `read32()` и используется только для чтения
   выровненных 4-байтных инструкций. После этого `read32()` почти ушел из горячего
   пути fetch-а.

4. `unordered_map` для блоков памяти был заменен на `vector<unique_ptr<Block>>`.
   Это убрало хэш-таблицу из верхушки профиля, но большого прироста по времени не
   дало, потому что после предыдущих оптимизаций основные расходы уже находились в
   `step()`, `execute()` и dispatch инструкций.

5. После этого был проверен вариант с разделением `step` на trace/non-trace путь.
   Первый вариант без принудительного inline оказался спорным: `perf report`
   показал отдельный вызов `stepImpl<false>()` внутри горячего цикла `run()`.
   После добавления `always_inline` время в большинстве benchmark-ов уменьшилось
   или почти не изменилось.

## Финальный результат

Финальный общий замер: `profile_res_after_run_step_rework_inline`.

| benchmark | before | after | result |
| --- | ---: | ---: | ---: |
| `prof_arith_loop` | 3.458 s | 3.0731 s | быстрее примерно на 11.1% |
| `prof_memory_loop` | 1.082 s | 0.57739 s | быстрее примерно на 46.6% |
| `prof_branch_loop` | 1.665 s | 1.07716 s | быстрее примерно на 35.3% |
| `prof_call_loop` | 0.971 s | 1.04599 s | медленнее примерно на 7.7% |
| `prof_ajmp_loop` | 5.904 s | 2.42698 s | быстрее примерно на 58.9% |
| `prof_page_alloc_loop` | 1.194 s | 0.7599 s | быстрее примерно на 36.4% |

Отдельно сравнивался вариант `stepImpl` до и после `always_inline`:

| benchmark | до always_inline | после always_inline |
| --- | ---: | ---: |
| `prof_arith_loop` | 3.4656 s | 3.0731 s |
| `prof_memory_loop` | 0.71627 s | 0.57739 s |
| `prof_branch_loop` | 1.5398 s | 1.07716 s |
| `prof_call_loop` | 0.9626 s | 1.04599 s |
| `prof_ajmp_loop` | 4.1457 s | 2.42698 s |
| `prof_page_alloc_loop` | 0.9197 s | 0.7599 s |

Также важно сравнение с предыдущим общим этапом `profile_res_after_vector_blocks`:

| benchmark | vector_blocks | run/step + always_inline |
| --- | ---: | ---: |
| `prof_arith_loop` | 2.958 s | 3.0731 s |
| `prof_memory_loop` | 0.765 s | 0.57739 s |
| `prof_branch_loop` | 1.537 s | 1.07716 s |
| `prof_call_loop` | 0.951 s | 1.04599 s |
| `prof_ajmp_loop` | 4.526 s | 2.42698 s |
| `prof_page_alloc_loop` | 1.055 s | 0.7599 s |

По этому сравнению видно, что разделение trace/non-trace пути не универсально:
`arith_loop` и `call_loop` стали хуже относительно предыдущего этапа, зато
`memory_loop`, `branch_loop`, `ajmp_loop` и `page_alloc_loop` заметно ускорились.

## Вывод

Самой полезной оптимизацией оказался уход от чтения инструкции через общий
`Memory::read32()`. Кэш блока тоже дал заметный эффект на memory benchmark-е.
Замена `unordered_map` на таблицу указателей логически убрала лишний поиск, но
по времени дала умеренный результат. Переработка `run/step` стала полезной только
после проверки через `perf report` и добавления `always_inline`, потому что без
него `stepImpl<false>()` не встроился в горячий цикл. Эффект этой оптимизации
получился неравномерным, поэтому её стоит оценивать по всему набору benchmark-ов.
