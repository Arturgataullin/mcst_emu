# Эмулятор процессора и ассемблер

Проект содержит ассемблер и эмулятор простого 32-разрядного процессора.
Процессор имеет 16 регистров общего назначения `R0` ... `R15`, а каждая
машинная инструкция занимает 4 байта.

Подробное описание регистров, бинарных форматов и семантики инструкций
находится в [docs/PROC.md](docs/PROC.md).

## Возможности

Ассемблер поддерживает:

- загрузку литералов: `LI`, `LUI`;
- арифметику: `ADD`, `SUB`, `MUL`, `UDIV`, `SDIV`;
- поразрядные операции: `AND`, `OR`, `XOR`;
- сдвиги: `SLL`, `SRL`, `SRA`;
- доступ к памяти: `LDB`, `LDH`, `LDW`, `STB`, `STH`, `STW`;
- преобразования данных: `SXT`, `BSWAP`;
- управление стеком и SCR: `SCRW`, `SCRR`, `ASPI`, `ASPR`;
- переходы: `RJMP`, `BRZ`, `BRNZ`, `AJMP`, `CALL`;
- сравнения: `EQ`, `NE`, `LT`, `GE`, `SLT`, `SGE`;
- псевдокоманды: `MOV`, `NEG`, `NOT`, `LFI`, `RET`;
- шестнадцатеричные литералы с префиксом `0x` или `0X`;
- комментарии, начинающиеся с `#`.

Эмулятор выполняет машинный код, проверяет индексы регистров и размер
программы, сообщает об ошибке при делении на ноль и выводит итоговое состояние
регистров. Память байтовая, little-endian и разреженная: блоки RAM по 4 КБ
выделяются по требованию.

## Структура проекта

- `common` - общие определения ISA и opcode;
- `assembler` - lexer, parser, lowering псевдокоманд, encoder и binary writer;
- `emulator` - разреженная память, загрузка программы и цикл fetch/decode/execute;
- `tests` - модульные, интеграционные и сквозные тесты;
- `docs/PROC.md` - спецификация процессора и системы команд;
- `examples` - [примеры программ](examples/README.md) на ассемблере.

## Пример

Файл `in.s`:

```asm
LFI R0 0x12345678
MOV R1 R0
NEG R2 R1
NOT R2
EQ R5 R0 R1
LI R3 0x0040
STW R0 R3 0x0
LDW R4 R3 0x0
```

Псевдокоманда `LFI` раскрывается в `LI` и `LUI`, `RET` раскрывается в `AJMP R1`,
а остальные псевдокоманды транслируются в существующие арифметические и логические инструкции.
`STW` и `LDW` показывают запись и чтение 32-битного слова из RAM.

## Требования

- компилятор с поддержкой C++20;
- CMake 3.20 или новее;
- GCC/Linux для целевого окружения задания;
- доступ к GitHub при первой тестовой сборке: Catch2 загружается через
  `FetchContent`.

## Сборка

Обычная сборка без тестов:

```bash
cmake -S . -B build
cmake --build build
```

Сборка с тестами:

```bash
cmake -S . -B build -DBUILD_TESTING=ON
cmake --build build
```

## Запуск

Ассемблирование программы:

```bash
./build/assembler/assembler in.s out.o
```

Выполнение бинарного файла:

```bash
./build/emulator/emulator out.o
```

Справка по опциям эмулятора:

```bash
./build/emulator/emulator --help
```

При успешном выполнении эмулятор печатает все пользовательские регистры:

```text
R0=0x12345678, R1=0x12345678, R2=0x12345677, ...
```

## Тестирование

Запуск всего набора тестов:

```bash
ctest --test-dir build --output-on-failure
```

Набор включает:

- unit-тесты lexer, parser и encoder;
- интеграционные тесты ассемблера;
- тесты памяти и выполнения инструкций эмулятором;
- сквозные тесты `исходный код -> ассемблер -> бинарный файл -> эмулятор`.

## Профилирование

Для профилирования используйте отдельную сборку `RelWithDebInfo`: в ней включены
оптимизации, но сохраняются имена функций для `perf` и `callgrind`.

```bash
cmake -S . -B build-profile -DCMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build build-profile
```

В `examples` есть длинные benchmark-примеры `prof_*.s`: арифметика, переходы,
`AJMP`, работа с RAM и `CALL`/`RET`. Их нужно запускать без трассировки, чтобы
измерять эмулятор, а не вывод в поток.

Собрать все `.s` файлы из директории в `.o` можно так:

```bash
./assemble_dir.sh examples
```

Для сбора `perf stat`:

```bash
./profile_perf_stat.sh profile_res_before
./profile_perf_stat.sh profile_res_before prof_memory_loop
```

Для `callgrind` с последующим `callgrind_annotate`:

```bash
./profile_callgrind.sh profile_res_before
./profile_callgrind.sh profile_res_before prof_memory_loop
```

Первый аргумент - директория для результатов. Второй аргумент необязательный:
если он не задан, обрабатываются все `examples/prof_*.s`; если задано имя
например `prof_memory_loop`, профилируется только этот пример.

## Трассировка исполнения

Трасса команд включается опцией `--trace=disasm`:

```bash
./build/emulator/emulator --trace=disasm out.o
```

Нумерация тактов начинается с `0`: номер такта равен количеству команд,
исполненных перед текущей командой.

В trace регистры назначения показываются после исполнения команды, а обычные
регистры-источники — до исполнения. Операнд SCR в `SCRR` выводится по текущему
значению, потому что сама команда `SCRR` его не изменяет:

```text
SCRW SP_TOP R5 (0xc000)
SCRR R6 (0xc000) SP_TOP (0xc000)
ASPI R6 (0xbff0) imm (0xfff0)
ASPR R6 (0xbff0) R5 (0xfffffff0)
```

Вывод можно ограничить включительными диапазонами тактов; пересекающиеся
диапазоны допускаются:

```bash
./build/emulator/emulator --trace=disasm --trace-ticks=1,3-7,5-10 out.o
```

Трасса записей в RAM включается режимом `ram-wr`:

```bash
./build/emulator/emulator --trace=ram-wr out.o
```

Можно включить несколько режимов сразу и ограничить RAM trace диапазонами
байтовых адресов:

```bash
./build/emulator/emulator --trace=disasm,ram-wr --trace-ticks=0-20 --trace-ram-addrs=64-95 out.o
```

`--trace-ram-addrs` фильтрует по байтовым адресам, затронутым записью. Если
невыровненная `STW` изменила две внутренние 32-битные ячейки, trace может
напечатать две строки для одной инструкции.

Обнуление освобождённого стека с `--clear-stack` также отображается в RAM trace
как запись, выполненная соответствующей командой `ASPI` или `ASPR`.

Поддержка трассировки включена при сборке по умолчанию. Отключить её можно
через `cmake -S . -B build -DTRACING=OFF`.

## Опции памяти и предупреждений

Размер RAM задаётся в байтах:

```bash
./build/emulator/emulator --ram-size=4096 out.o
```

Размер должен быть положительным и кратным 4. Для больших значений память не
выделяется целиком: блоки по 4 КБ создаются только при обращении к ним.

Освобождённые участки стека можно заполнять нулями в отладочном режиме:

```bash
./build/emulator/emulator --clear-stack out.o
```

Если одновременно включено предупреждение о неинициализированной памяти, освобождённые
байты снова считаются неинициализированными:

```bash
./build/emulator/emulator --clear-stack --warn=uninit-ram out.o
```

Предупреждения о чтении неинициализированных байтов включаются так:

```bash
./build/emulator/emulator --warn=uninit-ram out.o
```

Чтение байтов, в которые ещё ничего не записывалось, возвращает нули. Опция
дополнительно печатает предупреждение с тактом, адресом чтения и адресами
неинициализированных байтов внутри прочитанного диапазона.

После освобождения стека соответствующие байты всегда снова помечаются
неинициализированными. Без `--clear-stack` их прежние данные сохраняются, а с
`--clear-stack` чтение возвращает нули и одновременно выдаёт предупреждение.

## Ограничения

- поддерживаются только шестнадцатеричные числовые литералы;
- одна исходная инструкция должна находиться на одной строке;
- у инструкций формата `opcode a b imm8` третий операнд должен помещаться в
  диапазон `0x0` ... `0xff`;
- `NEG` и `NOT` используют скрытый временный регистр `R31` при раскрытии.
  Это соглашение реализации описано в `docs/PROC.md`.
